# GENERATE — 動画スライドのDesign用プロンプトを、ノウハウ参照で一気に生成する

## これは何か
「Week●の動画◯を作って」と指示されたら、私用ノウハウ(knowledge_engine/)をフルに参照し、本体リポジトリ(../edu-content-agents/)のカリキュラム・素材を踏まえて、Designにそのまま貼れる「Design用プロンプト」を一気通貫で生成するための指示書。

## 前提(ディレクトリ)
- 私用(このリポジトリ):ax-trainer-engine/ … knowledge_engine/ に教材設計のノウハウ(思想・視座・スケール・作法)がある。
- 本体:../edu-content-agents/ … カリキュラム設計、各Weekの素材、ブランド規定、品質原則、既存の完成物(samples/)がある。
- 2つは同じ親ディレクトリに並んでいる前提。本体から私用へは ../ax-trainer-engine/ で辿れる。

## Claude Codeへの指示:生成手順
「Week●の動画◯を作って」と指定されたら、以下を順に実行すること。

### ステップA:ノウハウを読む(私用)
knowledge_engine/ の4ファイルを必ず全て読み、内在化する。
- 01_educational_thesis.md(教育思想:受講者を担い手として扱う/自力でAIと向き合わせる/暗黙知の資産化)
- 02_perspective_converter.md(視座=変換器:事業視座で組む/フル注入の4点/メンバー視座に落とさない)
- 03_scale_design.md(スケール設計:動画①の縦のストーリー/欠落でなく進化に乗る/果実は断言しない)
- 04_slide_craft.md(作法:図解は軸を省かず立体的に/完全文/概念は具体先行/結論の帯/1スライド1主役/4層モデル厳守)

### ステップB:本体の文脈を読む
../edu-content-agents/ から、(a) 該当Week・該当動画のカリキュラム設計と射程、(b) 前後の動画との連続性、(c) brand_design.md、(d) slide_quality_principles.md(事業視座チェックリスト含む)、(e) samples/ の既存完成物(形式・品質の基準)を読む。

### ステップC:叩きを作る(ノウハウをフルに効かせる)
該当動画のスライド構成(5枚基準、ガイダンス回など必要なら6枚)を作る。このとき、knowledge_engine のノウハウを必ず反映する:
- 事業責任者の視座で書く。メンバー視座(作業手順の説明)に落とさない。
- 受講者を試さず・責めず、ハイパフォーマーの動作は「覗かせ、あなたも一部はやっている」共有の見せ方で。
- 結論を投げず、前提→なぜ→だから。価値観を揺さぶる情報を先に。差は具体で。
- 図解は軸を省かず立体的に。タイトルは概念の変化を置き、固有名・未習の専門用語を主役にしない。概念は具体先行。
- 4層モデル(フロー/アウトカム/タスク/プロセス)と既習の用語を正しく引き継ぐ。新概念を不用意に増やさない。
- スケールが要る回(特に各Weekの導入)は 03_scale_design.md の構造(進化に乗る/欠落でなく前進/果実は断言しない)を反映する。

### ステップD:Design用プロンプトに整形する
ステップCの叩きを、Designにそのまま貼れる1つのプロンプトに整形する。整形の形式は、本体の ../edu-content-agents/samples/week01/design_prompt_example_video4.md を実例テンプレートとして踏襲する。構成は:役割指定 → ブランド規定 → 設計原則 → 禁則事項 → この動画で起こす変化 → 設計の背骨 → スライド構成(S1〜) → 全体の補足。

- 出力するDesign用プロンプトは、共通ヘッダー(役割指定・ブランド規定・設計原則・禁則事項)と、その動画のスライド構成(この動画で起こす変化・設計の背骨・S1〜S5)を、1つの完結したプロンプトとして出力すること。共通ヘッダーを別ブロックに分離せず、動画1本ごとに、ヘッダー込みでそのままDesignに貼れる完結した1本のプロンプトにする。利用者が、ヘッダーと本体を別々に貼り合わせる手間が生じないようにする。

### ステップE:出力する
完成したDesign用プロンプトを、そのままコピーできる形(コードブロック)で出力する。あわせて、生成にあたり判断した点(例:スケールをどう設計したか、前動画からどう繋いだか)を数行で補足し、利用者が「通すか・直すか」を素早く判断できるようにする。

## 重要な制約
- 生成物(Design用プロンプト)は、利用者が本体側でDesignに使う「成果物」。一方、このGENERATE.md と knowledge_engine/ のノウハウ自体は、本体リポジトリには絶対に書き出さない・コピーしないこと。本体に残るのは、生成されたDesign用プロンプトと、その先の成果物だけ。
- コミット・プッシュは、明示的な指示があったときだけ行う。
